# Portable layer — run the AS suite on any MCP host

This directory holds the **generated adapters** that let the AS agent suite run outside GitHub
Copilot CLI — on Claude Code, Codex, Gemini CLI, Cursor, Goose, opencode, and any other
MCP-capable host — with the same agents, skills, lazy-loading, and quality gates.

Everything here is **generated from one source** (`registry/*.json` + `skills/INDEX.json`).
Don't edit by hand — edit the source and regenerate.

## Contents
| File | For | What it is |
|---|---|---|
| `CLAUDE.md` | Claude Code | instruction adapter (native skills + MCP) |
| `AGENTS.md` | Codex / opencode / Cursor / Amp | instruction adapter (MCP) |
| `GEMINI.md` | Gemini CLI | instruction adapter (MCP) |
| `mcp/claude.mcp.json` | Claude Code / Cursor | MCP server registration |
| `mcp/codex.config.toml` | Codex | MCP server registration |
| `mcp/gemini.settings.json` | Gemini CLI | MCP server registration |

## How the pieces fit
- **Tier-1 index** (`skills/INDEX.json`, generated by `scripts/gen_skill_index.py`) — the cheap
  always-on directory of skills.
- **Skills MCP server** (`immortals/skills_mcp.py`) — serves `skill_list` / `skill_get` /
  `skill_get_reference` so bodies load **only on demand**.
- **Adapters** (this dir, generated by `scripts/gen_adapters.py`) — tell each host what the
  agents are and how to use the lazy-load protocol.
- **Typed contracts** (`immortals/contracts/schemas/`) — the model-independent quality floor.

## Use it
1. Copy the instruction file for your host to your target project root
   (`CLAUDE.md` / `AGENTS.md` / `GEMINI.md`).
2. Register the MCP servers with the matching `mcp/` snippet (set `AS_SKILLS_DIR` if the
   `skills/` directory isn't co-located).
3. Point the host at the `agents/` you want (individually or in combos).

## Regenerate
```
python scripts/gen_skill_index.py     # refresh the Tier-1 index
python scripts/gen_adapters.py        # refresh these adapters
```
Both have a `--check` mode used as a CI gate (see `tests/test_prompt_lint.py`) so the adapters
can never drift from the source.

See `docs/cross-harness-tooling.md` for the full host landscape (OSS vs subscription) and the
portability optimization rationale.
